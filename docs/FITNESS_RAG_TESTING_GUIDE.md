# Fitness RAG Testing Guide

Короткая инструкция, как вручную проверить выполнение задачи “Первый RAG-запрос” через Android chat UI.

## Что проверяем

Нужно убедиться, что:

- fitness knowledge base проиндексирована
- чат работает в двух режимах: `Обычный` и `RAG`
- один и тот же вопрос даёт обычный ответ без retrieval и grounded-ответ с retrieval
- в режиме `RAG` виден блок `Knowledge Base Context`

## Prerequisites

Перед проверкой должны быть доступны:

- Android app
- локально запущенный `Document Index Server`
- corpus `demo/fitness-knowledge-corpus`

## Шаг 1. Поднять MCP серверы

В корне проекта:

```bash
./gradlew :mcp-server:runMultiServer
```

Что проверить:

- сервер document index доступен на `localhost:8084`
- Android эмулятор сможет ходить к нему через `10.0.2.2:8084`

## Шаг 2. Пересобрать индекс по fitness knowledge base

Во втором терминале:

```bash
bash scripts/reindex-fitness-knowledge.sh
```

Что это должно сделать:

- прочитать документы из `demo/fitness-knowledge-corpus`
- построить chunks для `fixed_size` и `structure_aware`
- сохранить индекс в локальный SQLite storage

Минимальный признак успеха:

- команда завершается без ошибки
- source для индексации: `fitness_knowledge`

## Шаг 3. Запустить Android app

Открой приложение на эмуляторе и перейди в экран чата.

Что должно быть видно:

- поле ввода сообщения
- переключатель режима ответа `Обычный / RAG`

## Шаг 4. Отправить канонический тестовый запрос

Основной пример запроса, который нужно отправить в чат:

```text
Что важнее для похудения: дефицит калорий или время приёма пищи?
```

## Шаг 5. Проверить режим `Обычный`

1. Выбери режим `Обычный`
2. Отправь канонический запрос

Что ожидать:

- чат должен вернуть обычный LLM-ответ
- блока `Knowledge Base Context` быть не должно
- ответ может быть общим и без явной привязки к corpus

## Шаг 6. Проверить режим `RAG`

1. Переключи чат в режим `RAG`
2. Отправь тот же самый запрос ещё раз

Что ожидать:

- чат должен вернуть ответ с retrieval
- должен появиться блок `Knowledge Base Context`
- в блоке должны быть видны:
  - source
  - strategy
  - chunks
  - title / section / score

Ожидаемая суть ответа:

- основным фактором должен быть назван дефицит калорий
- время приёма пищи должно быть описано как вторичный фактор или инструмент удобства
- ответ должен выглядеть более grounded, чем в режиме `Обычный`

## Критерии успеха

Задачу можно считать проверенной, если выполняются все пункты:

- MCP серверы поднимаются
- индекс `fitness_knowledge` пересобирается без ошибки
- чат отвечает в режиме `Обычный`
- чат отвечает в режиме `RAG`
- в `RAG`-режиме показывается `Knowledge Base Context`
- retrieved context содержит документы/секции из fitness corpus
- ответ в `RAG`-режиме опирается на corpus и ставит дефицит калорий выше тайминга питания

## Дополнительные запросы

Если нужен ещё один быстрый smoke-check, можно использовать:

- `Сколько белка обычно рекомендуют человеку, который хочет сохранить мышцы при похудении?`
- `Почему сон влияет на восстановление и контроль аппетита?`
- `Почему жидкие калории могут мешать снижению веса?`

## Дополнительно: evaluation runner

Если нужна не только ручная, но и полуавтоматическая проверка:

```bash
AI_API_KEY=... ./gradlew :mcp-server:runFitnessRagEvaluation
```

Runner:

- читает `demo/fitness-knowledge-corpus/fixtures/rag_questions.json`
- делает прогоны `PLAIN_LLM` и `RAG`
- сохраняет отчёты в `output/fitness-rag-evaluation/`
